# 维度 D6：可维护性与迭代支持

## 审查目标

评估代码的可维护性——下一个人能不能轻松理解和修改这段代码？

遗留系统的代码要活很久，可维护性决定了未来每次迭代的成本。

## 检查清单

### 1. 可读性

- 代码逻辑是否清晰易懂？
- 复杂逻辑是否有适当的分层或拆解？
- 变量名和函数名是否能自解释意图？
- 嵌套层级是否过深（> 3-4 层）？

### 2. 模块化程度

- 功能是否按职责合理拆分为独立模块/函数？
- 模块的边界是否清晰？
- 是否存在过大的函数（> 50 行逻辑）或过大的文件？

### 3. 扩展难度

- 如果未来需要在同一位置新增类似功能，改动范围有多大？
- 是否使用了硬编码而非配置化/参数化？
- 是否存在隐式约束只有当前作者知道？

### 4. 接口清晰度

- 函数的输入输出是否明确？
- 是否有隐式的副作用？
- 错误情况的返回是否有明确约定？
- API 契约是否对调用方友好？

### 5. 判定标准

可维护性不是"越抽象越好"——过度设计也是维护负担：
- **应标 FAIL**：300 行函数、5 层嵌套、无法从函数签名理解其行为
- **不应标 FAIL**：简单直接的实现，虽然没有用设计模式但清晰易懂
